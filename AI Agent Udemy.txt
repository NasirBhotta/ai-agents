CrewAI System Overview

CrewAI is an AI agent framework used to build and orchestrate multiple AI agents that can collaborate to solve tasks autonomously. The CrewAI ecosystem includes:

CrewAI Enterprise → a paid platform for deploying, monitoring, and managing agents.
CrewAI Studio → a low-code/no-code visual interface for building agent workflows.
CrewAI Framework → the open-source Python framework developers use to create AI agent systems.

The framework mainly supports two approaches:

Crews
A team-based autonomous system where multiple agents with different roles collaborate dynamically to solve complex problems. Best for creative, exploratory, and agentic tasks.
Flows
A structured and deterministic workflow system where tasks follow predefined steps and conditions. Best for controlled, auditable, and production-oriented processes.

CrewAI focuses on multi-agent collaboration, task delegation, and autonomous problem solving, making it useful for building advanced AI applications such as research assistants, analysis systems, customer support agents, and automated business workflows.



-> core concepts


	Framework for building multi-agent AI systems
	Uses Agents, Tasks, and Crews
	Agent → AI with role, goal, backstory
	Task → specific job given to an agent
	Crew → group of agents + tasks working 	together
	Execution types:
		Sequential → step-by-step tasks
		Hierarchical → manager AI assigns tasks
		More structured than OpenAI Agents SDK
		Supports YAML config to separate prompts from code


CrewAI + LiteLLM + Project Setup (Very Short)

CrewAI

Uses LiteLLM to connect easily to multiple models (OpenAI, Claude, Gemini, Grok, local models)
Model format: provider/model → easy switching between LLMs
Projects are fully structured (not single files)
Generated using: crewai create crew my_project
Project includes:
agents.yaml → agent configs
tasks.yaml → task configs
crew.py → agents + tasks + crew logic
main.py → entry point to run project
Run project using: crewai run
Separates config, logic, and execution for clean architecture





week 4 -> Langraph


-> agent workflows are represented as graphs
states -> represent the current snapshot of the application
Nodes -> Nodes are the python functions that represent agent logic. they reciece the current state and do something and return an updated state.
Edges-> are the python funtions that determine which Node to execute next based on the State. They can be conditional or fixed

nodes do the work
edges choose what to do next

five steps to create a graph
1=> define the state class
2=> start the graph builder 
3-> create node
4 -> create edges
6 => compile the graph

Annotated

Adds extra meaning to a type for LangGraph.
Example:

messages: Annotated[list, add_messages]

→ “this is a list + use add_messages on it”

LangGraph uses it to decide how state should behave.
→ tells it how to handle updates automatically


there is the reducer function in the langraph -> langgraph has nodes each node can have its own state so in order to keep the state persistent across each node a reducer is used
add_messages (Reducer)

Combines old + new values instead of replacing.
Example:

["Hi"] + ["How are you?"] → ["Hi", "How are you?"]
Prevents data loss in state updates.
→ keeps full chat history instead of overwriting it



super steps -> a super is the single iteration over the node. nodes that are parallel, that run parallel are the part of same super step.  nodes that run sequentially belong to seprate super step.

### LangGraph gives you tools to set the state back to a prior point in time, to branch off:

```
config = {"configurable": {"thread_id": "1", "checkpoint_id": ...}}
graph.invoke(None, config=config)
```

And this allows you to build stable systems that can be recovered and rerun from any prior checkpoint.


week 4 -> autogen

AutoGen = Microsoft's framework for building single-agent and multi-agent AI systems.
AutoGen Core = The underlying runtime that manages agent communication, events, and execution.
AutoGen AgentChat = The high-level framework used to create agents, conversations, tools, and workflows.
Models = The LLMs (e.g., GPT models) that power agent reasoning.
Messages = The fundamental communication unit; user inputs, agent responses, tool calls, and tool outputs are all messages.
Agent = An LLM with instructions, memory, and tool access that can perform tasks autonomously.
Team = A collection of agents collaborating toward a shared goal.
Tool = A function, API, database query, or external capability an agent can use.
Agent-to-Agent Communication = Agents can delegate work and exchange messages with other agents.
AutoGen Core vs AgentChat = Core provides infrastructure; AgentChat provides developer-friendly abstractions.
Studio = AutoGen's low-code/no-code environment for visually building agent systems.
Magentic-One = A prebuilt multi-agent application built on top of AutoGen.
AutoGen 0.4+ = A major rewrite introducing an asynchronous, event-driven architecture.
AG2 = A separate fork of the older AutoGen codebase maintained independently from Microsoft.
Microsoft AutoGen Focus = Enterprise-ready, scalable multi-agent systems.
Everything in AutoGen revolves around agents exchanging messages.
The main idea of AutoGen is coordination between multiple AI agents rather than a single chatbot.
For most developers, AgentChat is the part of AutoGen they interact with directly.
AutoGen Core is to agents what an operating system is to applications.
A Team is to AutoGen what a Crew is to CrewAI

on_messages() = Send messages to an agent and get its response.
Messages are passed as a list because AutoGen is conversation/message-oriented.
CancellationToken = A handle that allows an ongoing agent task to be cancelled.
CancellationToken() in examples usually means "not cancelling, just providing the required argument."
response.chat_message.content = The actual text generated by the agent.


AutoGen Core → Low-level framework that manages agent communication, routing, and lifecycle.
Runtime → The communication system that delivers messages between agents.
SingleThreadedAgentRuntime → A local runtime that processes all agent interactions in one process/thread.
Message → A custom data object used to exchange information between agents.
RoutedAgent → Base class that enables an agent to send and receive messages.
AgentId → Unique identifier of an agent consisting of type and key.
Agent Type → Describes the kind/category of agent (e.g., planner, worker, judge).
Agent Key → Identifies a specific instance of an agent type.
register() → Registers an agent type with the runtime so it can be created and used.
@message_handler → Marks a method as capable of receiving messages.
send_message() → Sends a message to a target agent and waits for a response.
MessageContext → Contains metadata about a message, including cancellation information.
Cancellation Token → Allows long-running operations like LLM calls to be stopped.
AssistantAgent → A ready-made AutoGen AgentChat agent powered by an LLM.
OpenAIChatCompletionClient → Connects an agent to OpenAI models.
OllamaChatCompletionClient → Connects an agent to locally running Ollama models.
Delegate Pattern → Wrapping an AssistantAgent inside a Core agent to handle LLM reasoning.
TextMessage → AgentChat's message format used by AssistantAgent.
Message Adapter → Conversion between Core Message and AgentChat TextMessage formats.
Agent-to-Agent Communication → Agents can directly send messages to other agents through the runtime.
Multi-Agent System → Multiple specialized agents collaborate to solve a task.
Judge Agent Pattern → One agent coordinates other agents and evaluates their results.
AutoGen Core Architecture → Message-driven architecture where agents communicate through a runtime.
LangGraph Architecture → State-driven architecture where nodes operate on shared state.
AutoGen Core vs LangGraph → AutoGen focuses on message passing; LangGraph focuses on state transitions.
Actor Model Similarity → AutoGen agents behave like actors that communicate via messages rather than shared memory.
The 5 most important interview-style points
Runtime is the post office of AutoGen Core.
Agents communicate through messages, not shared state.
Every agent has an AgentId(type, key).
@message_handler defines how an agent reacts to messages.
AutoGen Core is message-centric, while LangGraph is state-centric.

Distributed Runtime → Allows agents to communicate across processes and machines.
Host Service → Central message router and session manager.
Worker Runtime → Runs and manages actual agents.
gRPC → Network technology used for agent-to-agent communication.
Agent Advertisement → Workers tell the host which agents they contain.
Message Routing → Host finds the correct worker and forwards messages.
Standalone Runtime → Everything runs in one process.
Distributed Runtime → Agents can be spread across many servers.
Main Benefit → Scalability and separation of agents across machines.
Core Idea → Agents don't need to know where other agents are; the Host Service handles discovery and routing.
